# Feature: Clean up specs after OpenSpec migration

## Summary

Fix incorrect requirement mappings and missing technical requirements in the OpenSpec specs created during the initial migration. Add proper Purpose sections to every spec so the spec tree is self-contained and consistent.

## Motivation

The initial OpenSpec migration decomposed `REQUIREMENTS.md` into capability specs, but a review shows two categories of gaps:
1. UI requirements `UX04` and `UX09` were mapped to `collection-list` although they belong to `bgg-sync`.
2. Technical requirements `T03`, `T10`, and `T11` (local data persistence, thumbnail file cache, and schema versioning) are not explicitly covered.
3. All specs still contain a placeholder `## Purpose` section generated by the archive step.

This change resolves those gaps without changing any application behavior.

## Proposed Solution

- Update `REQUIREMENTS.md` capability index to reference the correct spec for `UX04` and `UX09`.
- Add new requirements to `openspec/specs/bgg-sync/spec.md` covering `UX04`, `UX09`, `T03`, `T10`, and `T11`.
- Replace placeholder `## Purpose` sections in all specs with meaningful descriptions.

## Alternatives Considered

- Keep the incorrect mappings and rely on cross-references. Rejected because the capability index is the canonical entry point and must be accurate.
- Create separate specs for persistence and schema versioning. Rejected because these concerns are intrinsically tied to sync behavior and are best documented in `bgg-sync`.
